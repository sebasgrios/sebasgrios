# 07 · Internationalization

## Locales soportados

| Code | Etiqueta | Por defecto |
|---|---|---|
| `es` | Español | ✅ |
| `en` | English | |

Añadir un locale en el futuro:
1. Añadir clave a `LOCALES` en `/src/config/i18n.ts`.
2. Añadir entrada en `/src/config/copy.ts` para los textos del chrome.
3. Añadir clave en cada `jsonb` traducible (data migration).
4. Actualizar constraint CHECK en cada tabla.

## Routing

Astro `i18n` config:

```ts
i18n: {
  defaultLocale: 'es',
  locales: ['es', 'en'],
  routing: {
    prefixDefaultLocale: false,
    redirectToDefaultLocale: false,
  }
}
```

Rutas resultantes:
- `/` → contenido en `es`
- `/en/` → contenido en `en`
- `/admin/...` no se localiza (solo el ingeniero).
- `/og/{locale}.png` para OG por idioma.

### Anchors

Anchors **siempre en inglés**, estables entre locales: `#experience`, `#education`, `#stack`, `#projects`, `#contact`. Los labels de la nav se traducen pero el href no cambia.

### Hreflang

Cada página incluye:

```html
<link rel="alternate" hreflang="es" href="https://sebasgrios.es/" />
<link rel="alternate" hreflang="en" href="https://sebasgrios.es/en/" />
<link rel="alternate" hreflang="x-default" href="https://sebasgrios.es/" />
```

## Patrón jsonb `{es, en}`

Tipo en TS:

```ts
export type Locale = 'es' | 'en';
export type Localized<T = string> = Record<Locale, T>;
```

Helper de resolución con fallback:

```ts
// /src/lib/domain/i18n.ts
export function pickLocale<T>(value: Localized<T> | undefined, locale: Locale, fallback: Locale = 'es'): T | '' {
  if (!value) return '' as T;
  return value[locale] ?? value[fallback] ?? '' as T;
}
```

Cualquier campo `jsonb` viene del repo como `Localized<string>` o `Localized<string>[]`. El componente que renderiza llama `pickLocale(field, locale)`.

## Fuente de `locale`

- En prerender: `Astro.params.locale` (si la ruta es localizada) o `Astro.currentLocale`.
- Helper único: `getLocale(Astro): Locale` en `/src/lib/i18n/getLocale.ts`. Si la URL es `/en/...` devuelve `en`, si no `es`.

## Copy del chrome

Lo que no vive en BD (eyebrows, botones, footer) se define en `/src/config/copy.ts`:

```ts
export const COPY: Record<Locale, ChromeCopy> = {
  es: {
    nav: { experience: 'Experiencia', education: 'Formación', stack: 'Stack', projects: 'Proyectos', contact: 'Contacto' },
    hero: { cta: 'Hablemos' },
    sections: {
      experience: { eyebrow: 'Trayectoria', title: 'Experiencia', sub: 'Casi cuatro años entregando producto real...' },
      // ...
    },
    footer: { madeWith: 'Hecho con ♥ y mucho café' },
  },
  en: { /* ... */ },
};
```

Tipo `ChromeCopy` es source of truth — añadir un nuevo idioma falla en compile time si falta una clave.

## Sitemap

`@astrojs/sitemap` con `i18n` config genera `<xhtml:link rel="alternate">` automáticamente.

## SEO

- `<html lang="es">` o `<html lang="en">` según ruta.
- `<title>` y `<meta description>` desde `profile.metaTitle`/`metaDescription` con `pickLocale`.
- `<meta property="og:locale" content="es_ES">` (o `en_US`).
- `<meta property="og:locale:alternate">` con el otro.

## Detección automática

**No** redirigimos automáticamente por `Accept-Language`. El usuario llega a la URL que pidió. La nav ofrece un selector ES/EN (link textual, no select).

Razón: el redirect automático rompe SEO y confunde al compartir links. Solución estándar (W3C) es ofrecer la alternancia, no imponerla.

## Pluralización / formato

- Fechas: `Intl.DateTimeFormat(locale, { month: 'short', year: 'numeric' })`. Helper `formatMonthYear(date, locale)`.
- Números: no hay (de momento).
- Pluralización: no hay (de momento). Si aparece, `Intl.PluralRules`.

## QA i18n

Cuando se añada/edita contenido en el backoffice, el form **obliga** a rellenar ambos idiomas (validación con Zod + UI). Si un campo se deja vacío en `en`, el helper `pickLocale` cae a `es` y se muestra una nota en la UI del backoffice (no en el portfolio público).
